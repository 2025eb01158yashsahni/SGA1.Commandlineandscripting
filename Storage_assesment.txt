----------------------------------------------------------------------
Using 'lsof', the following critical file hooks were identified for 
the application process:
* Regular Files: The application has active write-handles on main log files 
  (e.g., /var/log/nginx/access.log).
* Network Sockets: TCP/UDP streams are mapped as open file entities, 
  occupying communication channels.
* Shared Libraries: Multiple system .so files are held in read-only 
  memory maps by the process.

2. FILE DESCRIPTOR (FD) OBSERVATIONS
----------------------------------------------------------------------
Inspection of /proc/<PID>/fd revealed the standard structural breakdown:
* FD 0 (stdin): Linked to /dev/null (standard daemon isolation behavior).
* FD 1 (stdout): Pointed to the main application terminal output or log router.
* FD 2 (stderr): Pointed to error.log.
* FD 3+: Dedicated to specific log files, configuration resources, and active 
  network connections.

Critical Observation: If broken or dead symlinks appear here, it indicates 
that the application is attempting to write to a file that has been 
deleted or unlinked from the disk without restarting the process.

3. REDIRECTION RESULTS
----------------------------------------------------------------------
Simulated redirection testing yields the following behaviors:
* Standard Output ('>' or '1>'): Captures successful operations and clear-text 
  runtime metrics.
* Standard Error ('2>'): Successfully isolates execution anomalies, 
  uncaught exceptions, and access violations (e.g., "Permission denied").
* Combined Stream ('&>'): Merges both channels into a single unified timeline.

Diagnosis: Logging issues often arise when applications fail to properly 
separate stderr from stdout, causing critical faults to bypass traditional 
monitoring filters.

4. RESOURCE LIMITS OBSERVED
----------------------------------------------------------------------
* Session File Limit (ulimit -n): 1024 (Default standard)
* System-wide File Limit (fs.file-max): 9223372036854775807

Technical Risk: The default session limit of 1024 is highly restrictive for 
high-throughput web or application servers. Under heavy user load, the 
application will hit this threshold and trigger "Too many open files" errors, 
preventing it from opening new log streams or accepting network connections.

5. HOW LINUX MANAGES FILE I/O
----------------------------------------------------------------------
Linux unifies all system interactions under the core philosophy: "Everything 
is a file." 

When an application requests file interaction, Linux leverages a layered 
management hierarchy:

   [ Application Process ] 
             │
             ▼
   [ File Descriptor Table ]  <-- Process-specific index integers (0, 1, 2...)
             │
             ▼
   [ Open File Table ]        <-- System-wide table tracking offsets and modes
             │
             ▼
   [ VFS / Inode Table ]      <-- Virtual Filesystem maps to physical disk sectors

* File Descriptor (FD): A non-negative integer unique to each process. It 
  acts as a pointer to the kernel-level open file structure.
* System Open File Table: Tracks the read/write cursor offset, access mode 
  (read, write, append), and reference counts for all open files system-wide.
* Inode (Index Node): Represents the physical data block layout on the disk. 
  Even if an application changes its FD, the file's inode tracks its actual data.
